🔇 Audio Playback — Why Silent & Fix Plan

The user imported a video-with-audio clip. Video played. No sound. The entire audio subsystem exists (FFmpeg PCM decode → lock-free ring buffer → cpal CoreAudio output) but none of it fires because zero audio tracks exist on the timeline after import.

🔍 Root Cause

import_files() only creates video tracks and video clips. It never checks whether an imported asset has an audio stream, never creates an audio track, and never inserts an audio clip. The audio probe loop (app.rs:533) iterates audio tracks — finds none — and skips. The audio feed loop (app.rs:564) iterates audio tracks — finds none — and skips. Nothing ever calls audio.probe() or audio.feed_audio_at().

✅ The Fix (3 steps)

Step 1 — Detect audio: after import_media(), check the asset metadata for has_audio: true (already populated by probe_container_only() in pool.rs).
Step 2 — Auto-create audio track: if the file has audio and no audio track exists, call add_audio_track("A1").
Step 3 — Insert audio clip: create an audio clip with matching timeline_in and duration, linked to the video clip via link_group_id for grouped editing.

📐 What Already Works

Once audio tracks + clips exist, the rest of the pipeline is ready: audio.probe() reads metadata via raw FFI (~1ms), audio.feed_audio_at() starts async PCM decode (background thread), poll_completed() collects decoded PCM into cache, ring.write_samples() writes to lock-free buffer, cpal callback reads from buffer → CoreAudio output. Zero code changes needed in the audio subsystem itself.

• Video with multiple audio streams (multi-language) — select first stream or let user choose later.
• Audio-only files (mp3, wav, aac, flac) — create audio track + clip, no video track.
• Video with no audio — skip audio track creation (current has_audio flag handles this).
• Already has audio tracks — insert into existing A1 track (don't create duplicate).

🎯 Why avformat_find_stream_info Was the Hidden Freeze

🔍 The Misleading Probe

Using format::input_with_dictionary() with probesize limits looked like it should be fast — but ffmpeg-next's implementation always calls avformat_find_stream_info() after avformat_open_input(). That function opens every codec (avcodec_open2) and decodes sample frames to determine exact parameters — the exact same slow path as create_decoder(). The probesize options only limit how many packets are decoded, but the codec init itself still takes 1‑5s.

✅ The Raw FFI Fix

Dropped down to raw avformat_open_input() via ffmpeg_next::ffi. Skipped avformat_find_stream_info entirely. For MP4/MOV containers, the moov atom already carries codec parameters — width, height, codec ID, frame rate, and sample rate are all available directly from AVCodecParameters and AVStream without any codec initialization. For files where info is missing (e.g. raw TS streams), the function returns sensible defaults (1920×1080, 30fps).

📐 Three create_decoder() Calls Removed

Before this session, importing one video file triggered three separate create_decoder() calls — two in import_media() (probe + pool.open) and one in ensure_assets_open(). Each blocked for 1‑5s on the UI thread. Total beachball: 3‑15 seconds per file. After: zero. And the fourth call (the lazy open in decode_frame_rgba) now runs on a background thread.

What We Learned

🔍 Screenshot Pixel Analysis

Used screencapture → BMP → Python pixel scan to verify actual rendered output. Discovered that the theme was so dark that panel_fill(14,14,20) appeared as (27,27,27) in captures — indistinguishable from the preview's black(16,16,16) background. The timeline panel WAS rendering but its content wasn't visible because the entire app was nearly black.

⚠️ scroll_y ≠ scroll_x in ScrollArea

egui's recommended scroll-tracking pattern (ui.clip_rect - content_rect) works for horizontal scroll but can drift vertically when auto_shrink([false, false]) creates a viewport larger than the content. Critical Fix — BT.709 Color Matrix for HD Video

Fixed the most impactful one-file critical bug: wrong colors on all HD and UHD content. Every video loaded from a camera or phone showed incorrect saturation and crushed blacks. The root cause was two compounded errors in nv12_to_rgba() — wrong coefficient set and missing limited-range Y offset. The fix is 3 changed lines.

🎨 BT.601 → BT.709 Coefficients

rook-ui — panels/video_preview.rs

The conversion was using BT.601 matrix values (1.402 / 0.344136 / 0.714136 / 1.772) — correct only for SD/DVD content. HD and UHD content uses BT.709 (different primaries, different transfer). Replaced with correct BT.709 values: 1.79274 / 0.21325 / 0.53291 / 2.11240.

📐 Limited-Range Y Offset

rook-ui — panels/video_preview.rs

VideoToolbox is configured to output '420v' (kCVPixelFormatType_420YpCbCr8BiPlanarVideoRange) — Y in [16, 235], not [0, 255]. The old code used raw Y directly, compressing and offsetting every pixel's luminance. Fixed: yy = (Y − 16) × 1.16438 scales luma to the correct full output range.

Design Decisions

🟢 WASM is the recommended format

Sandboxed, cross‑platform, no native compilation. Plugin authors compile once to .wasm and it runs on macOS / Linux / Windows. wasmtime is pure Rust — no system libraries needed.

🟠 OFX for ecosystem compatibility

200+ commercial OFX plugins exist (Neat Video, Boris FX, Sapphire). Supporting OFX means Rook works with the professional plugin ecosystem from day one. Gate behind a feature flag for builds that don't need it.

🔒 Plugin isolation

WASM plugins cannot crash the host — wasmtime traps are caught as errors. OFX plugins run with a SIGSEGV handler: if a plugin crashes 3 times it gets auto‑disabled and a warning is shown in the browser.

📁 Filesystem‑based discovery

No online marketplace for v1. Plugins live in ~/.local/share/Rook/plugins/{wasm,ofx}/. Users drop files in and hit Refresh. PluginManifest is embedded in WASM custom sections or a sidecar .json for OFX.

🔗 Slots into existing effects

Plugins appear as EffectKind::Plugin(PluginId) — they use the same inspector, same keyframe system, same undo/redo. Zero new UI paradigms. A plugin is just another effect from the user's perspective.

📦 New crate: rook-plugin-host

Keeps wasmtime + libloading out of rook-core (which stays dependency‑free). rook-engine depends on it. If OFX is disabled at compile time, the OFX host compiles to a stub.

🔴 Still Broken — Workspace Layout

Root Cause: egui SidePanel Stacking

app.rs — right panel architecture

egui's SidePanel::right allocates width before the CentralPanel gets its share. Multiple SidePanels stack horizontally, each taking its default_width (220–280px). With 2–3 panels open the preview (CentralPanel) collapses to <100px — unusable. egui has no built-in "max total side-panel width" constraint. The Session 14 attempt to merge panels into one SidePanel with tabs only partially helped — the underlying issue is that egui's panel system is not designed for a pro video editor layout where the preview must always dominate.

Why the Tabbed Panel Didn't Fix It

Single SidePanel still competes with CentralPanel

Even a single SidePanel::right at 240px steals width from the preview. On a 1280px-wide window, that leaves ~1040px for the preview — fine. But when the inspector content is tall and the user resizes the panel to see more, the preview shrinks further. egui's layout model gives SidePanels priority; the CentralPanel gets the remainder. There is no way to say "CentralPanel must be at least 640px; SidePanels must yield."

Scrolling Still Flaky

timeline.rs — ScrollArea vs manual offset tracking

The pending_scroll approach reduces the fight between our manual scroll_x/scroll_y and egui's ScrollArea internal state, but the hand-tool panning has a one-frame lag and trackpad scroll sometimes jumps. The fundamental problem is mixing programmatic scroll control (Fit, F-key, Hand tool) with egui's built-in ScrollArea state management. A cleaner approach is needed — either fully embrace egui's ScrollArea state (read-only, no manual offset) or bypass it entirely and implement custom scrolling.

🏗️ Key Architectural Decision

Drop egui Panels for the Main Workspace

SidePanel/CentralPanel → manual rect allocation

Why: egui's panel system allocates space with SidePanels taking priority. There is no way to guarantee the CentralPanel a minimum width. For a video editor where the viewer must always be prominent, this is a deal-breaker. We need manual control over column widths.

How: In RookApp::update(), after the menu bar and before the timeline, allocate the remaining space with ui.allocate_rect or nested ui.horizontal with fixed ratios. Compute three column rects with the viewer guaranteed ≥640px. Pass each rect to the appropriate panel's show() method.

Risk: We lose egui's built-in panel resizing and collapsing. We'll need to implement drag-to-resize dividers manually (tracking pointer drag on a narrow "gutter" rect between columns). This is ~100 lines of well-understood immediate-mode GUI code.

Keep TopBottomPanel for Menu + Timeline

Only the middle section needs the custom layout

The menu bar (TopBottomPanel::top) and timeline (TopBottomPanel::bottom) work fine as egui panels because they take the full width and a fixed height. Only the middle section (browser + viewer + inspector) needs the custom manual layout. This limits the scope of the change to ~200 lines in app.rs.
